## Problem Statement

逛逛AI是一个AI穿搭推荐电商产品的教学Demo，用于向"AI原生产品经理工作流"课程的学员展示：一个大模型驱动的产品从用户需求洞察到原型交付的完整链路。目前已有Google Stitch生成的两套静态原型（用户端5屏+后台4屏），需要将它们转化为一个可以真正交互、可以部署、学员扫码就能在手机上体验的React Web App。

核心目标：让学员看到一个**跑起来的AI原生产品**，而不是一套静态截图。

## What Makes This Cool

- **场景化AI穿搭的完整闭环**：选场合 → AI出方案 → 看详情 → 收藏/下单 → 分享穿搭卡片，全流程可演示
- **用户端+后台双视角**：用户端展示C端体验，后台展示产品经理如何用数据驱动迭代——两个视角合成"AI产品全貌"
- **设计系统完整**：已有一套成熟的"Soft Professionalism"设计语言（Soft Coral Pink + Plus Jakarta Sans + Warm Grey-White），Demo呈现出接近真实产品的视觉品质
- **教学价值**：这个Demo本身就是01-11模块全套方法论的产出物——从用户调研（01-02）到竞品分析（03）到需求整理（04-05）到产品创新（06）到价值评估（07-08）到项目计划（09）到PRD（10）到原型生成（11）

## Constraints

1. **时间**：黑客松/Demo节奏，需要快速出成品
2. **形态**：Web App，手机端优先（学员扫码体验），桌面端适配后台
3. **数据**：全部使用Mock数据，不需要真实AI接口
4. **现有资产**：9个Stitch HTML文件 + 2套完整DESIGN.md设计系统
5. **交付物**：可部署的React项目 + 在线可访问URL
6. **教学场景**：课堂上老师展示，学员扫码体验，需要稳定可靠

## Premises

1. 教学Demo的核心价值是展示"完整闭环"和"产品方法论"，而非AI算法精度 — 模拟数据足够
2. React SPA是最佳形态——工程结构正经、可扩展、学员看到的是"正式项目结构"
3. 现有设计系统（DESIGN.md）直接作为React项目的设计Token来源
4. 用户端和后台分别独立SPA，各自维护独立mock数据——两个App展示同一产品的不同视角，不需要实时联动
5. Mock层与真实接口保持相同的数据结构（TypeScript类型），后期接入真实API时替换mock handler即可

## Approaches Considered

### Approach A: 原型增强
直接在现有Stitch HTML上加JavaScript交互，纯静态文件部署。最快但不可扩展。

### Approach B: React SPA + Mock Data（✅ 选定）
用React重建用户端5屏+后台4屏，各自独立SPA，mock数据内嵌。正经工程结构，部署到Vercel。

### Approach C: 教学工具箱
在方案B基础上嵌入教学标注层、Prompt演示、模块映射标注。教学价值最高但工作量最大。

## Recommended Approach

**方案B：React SPA + Mock Data**

### 技术栈与架构决策

| 决策点 | 选择 | 原因 |
|--------|------|------|
| 框架 | React 18 + TypeScript | 生态成熟，学员最可能接触 |
| 构建 | Vite | 快、零配置 |
| 样式 | Tailwind CSS + DESIGN.md token | 设计系统直接映射为theme config |
| 路由 | React Router v6 | SPA标准，支持URL参数（occasion等） |
| 状态管理 | React Context + useReducer（每App一个Store） | 5屏不需要Redux |
| Mock数据 | 内嵌在Context初始值中，localStorage持久化 | 简单直接，无外部依赖 |
| 图表（后台） | Recharts | 轻量、React原生 |
| Canvas降级 | html2canvas（可选依赖） | 仅在Canvas API失败时使用，按需加载 |
| 部署 | Vercel（两个独立project，各一个域名） | 免费、零配置 |
| monorepo工具 | pnpm workspaces | 简单、快、原生支持workspace协议 |

### 用户端页面与状态矩阵

| 页面 | 路由 | Loading态 | Empty态 | Error态 | 核心交互 |
|------|------|-----------|---------|---------|----------|
| 首页 | `/` | 骨架屏（场合卡片占位4个） | "暂无推荐"引导设置偏好 | 隐藏（纯静态数据不会出错） | 场合卡片点击→推荐页 |
| 推荐结果 | `/recommend/:occasion` | 穿搭卡片骨架屏（3张） | "该场合暂无推荐方案"带引导 | "加载失败，请返回重试" | 场合横滑切换；价格筛选；收藏/分享 |
| 风格偏好 | `/preferences` | 无需（表单即显） | 首次使用需引导填写 | "保存失败"（localStorage满时） | 颜色/预算/风格选择；保存反馈 |
| 收藏/历史 | `/favorites` | 列表骨架屏 | 空状态插画+"去逛逛"CTA按钮 | 隐藏 | 查看收藏；取消收藏；浏览历史 |
| 分享图 | `/share/:outfitId` | Canvas生成loading | 穿搭ID不存在时显示404提示 | Canvas生成失败时降级为纯文本 | 预览穿搭卡片；保存图片 |

|------|------|-----------|---------|---------|----------|

> **决策：去掉"系统设置"页面。** 推荐参数调优和品牌库管理是生产环境功能，教学Demo不需要，且在mock环境下无实质教学价值。
>
> **Toggle说明**：后台启用/禁用toggle仅改变后台表格的视觉状态（灰色标记已禁用项），不跨App影响用户端数据——两个SPA各自维护独立mock数据。生产环境需共享backend实现跨端同步，这个架构差异本身就是教学讨论点。

### 关键交互流

```
[用户端 —— 完整闭环]
首页 (/)
  → 点击场合卡片"上班通勤"
  → 跳转 /recommend/work-commute（推荐页，显示6套上班通勤穿搭）

    → 价格筛选"¥200-500" → 列表过滤为3套
    → 点击某套穿搭 → 展开详情面板（单品列表+价格明细+品牌信息）
      → 点击心形收藏 → 图标填充+toast提示"已收藏"（写入localStorage）
      → 点击分享按钮 → 跳转 /share/outfit-work-001
        → Canvas绘制穿搭卡片（主图+单品缩略+场合标签+总价）
        → 点击"保存图片" → Canvas转Blob → download
        → 点击"返回" → 回到推荐页

[后台 —— 独立数据查看]
Dashboard (/)
  → 显示预填充的mock指标：今日营收¥12,580 | 活跃用户342 | 订单89 | 收藏156
  → 热门场合TOP3：上班通勤(42%) / 周末约会(31%) / 闺蜜聚会(18%)

  → 偏好分布饼图：简约通勤(38%) | 优雅知性(29%) | 潮流街头(18%) | 温柔甜美(15%)
  → 颜色偏好柱状图
```

> **设计决策：用户端与后台独立。** 两个SPA各自维护自己的mock数据集。不需要跨App实时联动（那需要共享backend或BroadcastChannel，对教学Demo过度工程）。老师在课堂上展示时，可以分别在两个浏览器tab打开用户端和后台，手动对比——这本身就是教学材料："做产品要看数据，后台就是你的仪表盘。"

### 图片素材策略

- **来源**：使用本地静态图片（`assets/outfits/`），约20-30张
- **获取方式**：从Unsplash（免版权时尚摄影）或Pexels下载，统一裁剪为3:4比例
- **命名规则**：`{outfit-id}-main.jpg`（主图），`{outfit-id}-item{N}.jpg`（单品图）
- **AI生成备选**：如果手动找图太慢，用Midjourney/DALL-E生成统一风格的服装搭配图（prompt: "minimalist fashion flatlay, soft coral pink background, office casual outfit, clean product photography style"）

### Canvas分享图规格

- 尺寸：750×1334px（iPhone 6/7/8比例，适合微信分享）
- 布局：顶部主图（70%高度）→ 底部信息栏（穿搭名称+场合标签+总价+小程序码占位）
- 降级方案：Canvas失败时，用纯CSS+HTML的卡片截图替代（html2canvas库）
- 依赖：仅使用原生Canvas API，不加额外库。如需要下载用`canvas.toBlob()`
- DPR处理：画布逻辑尺寸750×1334，实际像素需×`window.devicePixelRatio`缩放，否则Retina屏模糊
- 图片跨域：本地图片无跨域问题；若未来使用远程图片，需设置`img.crossOrigin = 'anonymous'`

### localStorage 初始化策略

每个App的Context初始化时：
1. 优先从localStorage读取已有数据
2. 若localStorage为空（首次访问或用户清除），使用硬编码的种子数据（seed data）
3. 种子数据写入localStorage，后续操作以localStorage为准
4. 用户操作（收藏、偏好修改等）即时同步写入localStorage
